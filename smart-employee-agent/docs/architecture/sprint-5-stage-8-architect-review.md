## Verdict

**GO-WITH-CHANGES.**

The "LLM = router/composer, never an authority" boundary is sound and rests on real, already-shipped enforcement points (server-fixed `_TOOL_REGISTRY` scopes + IS CIBA denial + `_REQUIRED_ARGS` pre-CIBA check + `textContent` rendering). The fallback design is coherent; every Gemini failure mode degrades to Sprint-4 behaviour rather than erroring. The `chat/routes.py` refactor is genuinely minimal. The slice ordering is correct with no dependency inversion. None of the findings below are architectural blockers — they are spec-tightening items that should be folded into the slices before/while they are built. Two of them (F-1, F-2) must be resolved in the doc set or the very first commit because they affect the security-invariant exit criteria; the rest are non-blocking.

### Rationale

1. **The authority boundary is real, not aspirational.** Trace from `resolve_tool_calls` → `ToolCall(agent_id, tool_id, args)` → `_run_serial_fan_out` → `client.message_send(token_a, tool_id, args, …)` → `HRDispatcher.__call__`: the dispatcher looks the tool up in `_TOOL_REGISTRY` (`registry_entry = _TOOL_REGISTRY.get(tool)`), takes `tool_scope_override` from that tuple, and that — not anything the LLM said — is what is sent to `ciba_client.initiate(scope=ciba_scope)`. The LLM's `args` dict only ever reaches `kwargs_builder(args)`, which whitelists keys per tool. An LLM that picks `hr.cubicle_assign` for an `Employee` gets `openid hr_assets_write_rest` requested at IS, IS denies, `ERR-CIBA-005`/`ERR-MCP-003` comes back, user sees "not authorised." There is no code path in v4 where an LLM token (router output or composer output) is read as a scope, a consent string, an `aud`, or a `sub`. The `args`-filtering step in `_validate` (strip hallucinated keys, scalars only) is the right belt over the suspenders.

2. **Consent cannot be bypassed.** Consent is structural: every `_TOOL_REGISTRY` entry yields a `ConsentRequiredPayload` unless the per-`(user_sub, scope)` token cache has a live token (UC-06/D2.5) — and that cache is keyed on the *server-fixed* scope, not the LLM call. The LLM cannot pre-warm or skip the cache. `action_text` on the widget is built dispatcher-side from `_TOOL_REGISTRY` + sanitised tool args (`_sanitise_action_text`, F-08 charset/length cap), never from LLM output.

3. **The fallback is complete.** Truth table §10 covers all four cells of `{router ok|fail} × {composer ok|fail}` plus the keyword-only and the prompt-injection rows; `resolve_tool_calls` catches `LLMError` and falls to `deps.keyword_router.route(...)`, the composer wiring catches `LLMError` and falls to `"\n\n".join(per_tool_outputs)`, and `GeminiLLMClient.route/compose` wrap *everything* (`except (asyncio.TimeoutError, Exception)`) in `LLMError`. The one genuinely-missing branch is small (F-3, below): the "router OK, but the LLM returned `[]` *and the keyword router is also empty*" path — the spec is internally inconsistent about what the user sees there, but it never errors.

4. **The card⇔registry contract is the right shape for a demo, with one caveat.** Making `Skill.args` the source of the arg-name list the LLM is shown, and adding `test_agent_card_matches_dispatcher_registry` to fail on drift, is a reasonable contract: the card is the public projection, the dispatcher is the authority, the test is the lint that keeps them in sync. The caveat (F-4): `scope` already lives on `Skill`, and `_TOOL_REGISTRY` has its own (sometimes different — `openid …`-prefixed, write-tier-overridden) scope. The consistency test must assert *only* the tool-id set, not scopes, or it will produce false drift. Document that explicitly.

5. **Slice order is correct.** S5.1 (`apply_leave`, zero LLM dependency) → S5.2 (config + client + router + card expansion) → S5.3 (composer) → S5.4 (SPA + compose passthrough + demo-up). No inversion: S5.2 needs the config/client S5.0 folded into it; S5.3 needs the `outcomes` list which S5.2 only threads (doesn't consume); a half-built S5 at any checkpoint degrades to `_render_result`. S5.1-first is the right de-risking call (closes the long-open UC-13 gap independently).

6. **Latency mitigation is adequate for a demo.** Two `gemini-2.5-flash` round-trips with `temperature=0/0.3`, ≤512 output tokens, ≈1–2 KB prompts, 8 s hard timeout → fallback each: ≈1–3 s added per turn, with a "Thinking…" affordance covering the router leg. A combined router+compose call is correctly *not* pursued — the composer needs the tool *results*, which only exist after the serial fan-out, so the calls genuinely cannot be merged without restructuring the fan-out. Don't pursue it now.

## Findings

Severity scale: **HIGH** (security-invariant or correctness risk; resolve in docs or first commit) · **MEDIUM** (will bite during implementation if left implicit) · **LOW** (polish / nice-to-have).

### F-1 — HIGH (blocking) — `_validate` must reject LLM-emitted args for `_REQUIRED_ARGS` write-tier tools *only* by name, not by value; spec is silent on the dangerous middle case

**What:** `_validate(routed, deps)` (tech-arch §5) filters `args` to "keys present in that skill's `args` list" and "JSON scalars only." Good. But there is no statement about what happens when the LLM emits a *valid scalar value for a required arg that it hallucinated* — e.g. for `hr.approve_leave` the LLM invents `{leave_id: "LV-001"}` when the user never named a leave id, or for `hr.cubicle_assign` it invents `{cubicle_id: "C-099", employee_username: "everyone"}`. The `_REQUIRED_ARGS` pre-CIBA check (`hr_agent/ciba/orchestrator.py` line ~421) only checks *presence*, so a hallucinated-but-present value sails straight to CIBA. This is not a *scope* escalation (the scope is still server-fixed and IS still gates the role), and it is not strictly new (the keyword router's `_extract_inline_args` also pulls `LV-\d+` regex matches), but the LLM is a far more prolific fabricator of plausible IDs than a regex. The Stage-11 prompt-injection test case (#4) tests *scope* escalation, not *fabricated-target* escalation.

**Where:** `orchestrator/llm/router.py` `_validate` (new, S5.2); the gap is the interaction with `hr_agent/ciba/orchestrator.py:_REQUIRED_ARGS` presence-only check; UC-17 EX-2 / Stage-11 test #3 cover the *missing*-arg case but not the *fabricated*-arg case.

**Recommended fix:** (a) State explicitly in the tech arch / UC-17 that for write-tier tools (`hr.approve_leave`, `hr.reject_leave`, `hr.cubicle_assign`, `it.issue_asset`) the consent widget's `action_text` is the user's last line of defence — it names the cubicle/leave-id/recipient that the LLM picked, rendered verbatim from the args, so the human approver sees "Assign cubicle C-099 to everyone" and declines. This is already true in code (`_sanitise_action_text` path in `HRDispatcher.__call__`) — just make it a documented invariant, not an accident. (b) Add a Stage-11 manual-gate case: "ask the agent to approve/assign something for a target you never mentioned → the consent widget shows the fabricated target → you decline → no write." (c) Optional, non-blocking: have the composer prompt + the dispatcher's `ERR-AGENT-002` reason text be the only place a *missing* required arg surfaces — never let `_validate` "helpfully" inject a default.

### F-2 — HIGH (blocking) — `OrchestratorConfig` is `frozen=True, slots=True`; the `from_env` warn-don't-crash path for `llm` + no key is fine, but the new fields must be added to the dataclass body *and* the `cls(...)` call, and `main.py` must build `llm_client=None` in that branch — the docs say this but the failure mode if it's missed is a silent keyword-only run that *looks* like a Gemini outage

**What:** Not a design flaw — a "this will bite if implemented carelessly" item with security-adjacent consequences. The plan (tech-arch §7, slice S5.2) correctly says: `from_env` warns (doesn't crash) if `LLM_FALLBACK_MODE=llm` but `GEMINI_API_KEY` empty; `main.py` then builds `llm_client=None`; `resolve_tool_calls` / `compose_reply` both no-op to keyword. The risk: `OrchestratorConfig` is `@dataclass(frozen=True, slots=True)` — adding `gemini_model`, `llm_timeout_s`, `llm_max_output_tokens` requires touching *three* places (the field declarations with defaults, the `from_env` parse block, and the `return cls(...)` kwarg list) or `from_env` raises `TypeError` at boot — which is a *crash*, not graceful degradation, contradicting the §2.6 / exit-criterion-5 promise. Also: `llm_fallback_mode` is currently parsed without `.strip() or "keyword"` normalisation (`config.py` line 281: `env.get("LLM_FALLBACK_MODE", "keyword").strip()` — note: an explicit empty `LLM_FALLBACK_MODE=` yields `""`, which is falsy and means `use_llm` is correctly False, but it's worth the `or "keyword"` for symmetry with the tech-arch §7 snippet).

**Where:** `orchestrator/config.py` (`OrchestratorConfig` dataclass body + `from_env` + `cls(...)` call), `orchestrator/main.py` (the `if cfg.llm_fallback_mode == "llm" and cfg.gemini_api_key:` branch).

**Recommended fix:** In the S5.2 slice checklist, make "add the three fields to the dataclass body, the parse block, *and* the `cls(...)` kwargs" an explicit sub-step (it's easy to add to two of three). Add a `test_config_from_env_llm_mode_no_key_does_not_crash` to S5.2's test list (it's mentioned generically as "config `from_env` (mode/key/new vars)" — name the no-key-no-crash case specifically). Add `or "keyword"` to the `llm_fallback_mode` parse for parity with the documented snippet.

### F-3 — MEDIUM (non-blocking) — "router OK, LLM returned `[]`, keyword router also empty" — UC-17 EX-1 and tech-arch §6 disagree on what the user sees

**What:** UC-17 Main-flow step 4 + EX-1 step 3 say: if both the LLM and the keyword router produce no tool calls, "the orchestrator replies with the LLM's natural-language clarification text (step 8) or a generic 'I'm not sure what you'd like me to do.'" But step 8 *is* the composer, and the composer's contract is `compose(user_message, outcomes)` — with zero tool calls there are zero `outcomes`, `_run_serial_fan_out` is never invoked, and `post_chat`'s existing path just emits the hard-coded `ChatMessageEvent(content="I don't know how to help with that.")`. So there is no "LLM clarification text" in this branch — the router prompt explicitly tells the LLM to return `[]` for chit-chat/off-topic and the router never asks the LLM for prose. This isn't a hard error (the user gets the generic line), but the doc set promises something the design doesn't deliver, and an implementer will either (a) build a third LLM call (scope creep, contradicts §4 "single-shot per message") or (b) leave the generic line and quietly diverge from the UC.

**Where:** `orchestrator/chat/routes.py:post_chat` (the `if not tool_calls:` branch); UC-17 §Main-flow step 4 + EX-1 step 3-4; tech-arch §6 (`post_chat` "already handles that earlier with the 'I don't know' ChatMessageEvent").

**Recommended fix:** Pick one and write it down. Recommended: keep the generic line for the empty-empty case in S5 (no third LLM call), and either (i) delete the "LLM's natural-language clarification text" sentence from UC-17 step 4 / EX-1, or (ii) if the product genuinely wants a friendlier "I can help with leave, cubicles, and IT assets — what would you like?" line, hard-code *that* (it's a fixed string, not LLM output) — it's a UX copy change, not an architecture change, and it belongs in Stage-4 UX, not here. Note this is also where the §4 "stretch goal: chat-history replay" would naturally help (a chit-chat turn could be handled conversationally) — but that's explicitly out of the S5 commitment, so don't let it leak in via this branch.

### F-4 — MEDIUM (non-blocking) — `test_agent_card_matches_dispatcher_registry` must assert tool-id *set* only, not scopes; and the `Skill.args` field needs a validator or the test will paper over typos

**What:** Two sub-issues with the new contract. (1) The Stage-10 test (`{s.tool_id for s in card.skills} == set(_TOOL_REGISTRY)`) is the right assertion *for tool ids*. But `Skill` already has a `scope` field, and several `_TOOL_REGISTRY` entries carry a *different* scope string than the card (the registry uses `"openid hr_basic_rest"`, `"openid hr_assets_write_rest"` etc. — the `openid`-prefixed CIBA scope; the card uses the bare `hr_basic_rest` resource scope; `hr.read_balance` in the registry has `scope_override=None` and inherits `deps.ciba_scope="openid hr.read"` which doesn't match the card's `hr_self_rest` at all). If the consistency test ever grows a scope assertion it will fail spuriously. State in tech-arch §9 that the test asserts the tool-id set *only*; the card's `scope` is documentation/LLM-context, the registry's is authority, and they are deliberately not required to match. (2) `Skill.args: list[str]` with no validator means a typo in a fixture (`"start_dat"` instead of `"start_date"`) silently produces a catalogue entry the LLM is told about, the LLM dutifully omits/includes the wrong key, `_validate` strips it as "not in the skill's args list" — wait, no: `_validate` keeps keys *in* the skill's args list, so a typo'd skill arg means the LLM's *correct* `start_date` gets *stripped* and the dispatcher then sees a missing required arg → spurious `ERR-AGENT-002`. The Stage-10 consistency test as specced (tool-id set only) does *not* catch this.

**Where:** `orchestrator/agent_registry/cards.py` (`Skill` model — new `args` field, S5.2); `common/a2a/agent_card.py` if `Skill` is the shared model (it is — `agent_card.Skill`, not `cards.Skill`; tech-arch §1 says "`orchestrator/agent_registry/cards.py` — `Skill` model gains optional `args`" which is *wrong about the location* — `Skill` lives in `common/a2a/agent_card.py`); the Stage-10 test.

**Recommended fix:** (a) Correct tech-arch §1: the `Skill.args` field is added to `common/a2a/agent_card.py:Skill`, and `llm_projection()` (same file) must surface it — `cards.py:llm_tool_list()` just re-exposes what `llm_projection` returns. (b) State in §9 that the consistency test asserts the tool-id set only. (c) Strengthen the consistency test (or add a second one) to also assert, for each `_TOOL_REGISTRY` entry that has `_REQUIRED_ARGS`, that *every required arg name appears in the card skill's `args` list* — this is the assertion that actually protects against the typo-strips-the-right-key failure. Cheap, deterministic, no LLM. (d) Decide whether `it.list_available_assets`'s `asset_type` (an *optional* dispatcher arg, not in `_REQUIRED_ARGS`) goes in the card `args` — recommend yes, so the LLM can use it; the contract should be "card `args` ⊇ everything the kwargs_builder reads," not just the required subset.

### F-5 — MEDIUM (non-blocking) — `_run_serial_fan_out` building `outcomes` alongside `per_tool_outputs` doubles the number of places that must stay in sync; the `ToolOutcome` mapping for the agent-not-in-registry / no-A2A-client / unexpected-payload branches is unspecified

**What:** The refactor (tech-arch §6) says: as the fan-out loops it builds *both* `per_tool_outputs: list[str]` (the fallback text fragments, exactly as today) *and* `outcomes: list[ToolOutcome]` (for the composer). On a `ResultPayload`: `ToolOutcome(ok=True, data=...)`. On `ErrorPayload`/CIBA-error/denied/expired/unexpected: `ToolOutcome(ok=False, error_id=..., reason=...)`. But `_run_serial_fan_out` has *several* early-`continue` branches that only append a `per_tool_outputs` string and have no obvious `error_id`: `card is None` (appends "I could not reach the agent…"), `client is None` (appends "I could not connect…"), `message_send` raised (`A2AError`/`Exception`, appends "I could not reach {agent_label}…"), `await_completion` raised (appends "There was a problem completing consent…"), and the two "Received an unexpected response" guards. If those branches forget to also append a `ToolOutcome`, the composer gets a *shorter* outcomes list than the number of tools that ran, the prompt rule "mention the outcome of every tool" silently can't be satisfied, and the user sees a reply that drops a tool with no error. Not a *security* issue — but it's exactly the kind of "two parallel accumulators, one updated in N branches" pattern that rots.

**Where:** `orchestrator/chat/routes.py:_run_serial_fan_out` (all the `continue` branches); tech-arch §6 (which only enumerates the `ResultPayload` and `ErrorPayload`/CIBA cases).

**Recommended fix:** Two options, recommend the first: (a) **Single accumulator.** Build *only* `outcomes: list[ToolOutcome]` in the loop; derive `per_tool_outputs`/`fallback_text` from it at the end by mapping each `ToolOutcome` through the existing `_render_result` / `_friendly_error` (success → `_render_result(agent_label, tool_id, ResultPayload(data=...))`; failure → `_friendly_error(error_id, reason, agent_label)`). The "agent not in registry" / "no A2A client" / "transport error" branches map to a `ToolOutcome(ok=False, error_id="ERR-AGENT-002" / "ERR-AGENT-001", reason=...)` — which is *more* honest than today's bespoke strings anyway. This makes "every tool that ran produces exactly one outcome" a structural guarantee, not a discipline. It is *not* "minimal" in the line-count sense but it is minimal in the invariant-surface sense, and the §6 refactor is already touching every one of those branches to thread `user_message` — fold the accumulator change in. (b) If you keep two accumulators (truly minimal diff): the tech arch *must* enumerate the `ToolOutcome` for *every* `continue` branch, and the S5.3 test list *must* include "fan-out where the first tool's agent is unregistered → composer receives a 2-element outcomes list (one error, one ok)." Don't leave the mapping implicit.

### F-6 — LOW (non-blocking) — `apply_leave` MCP-tool scope: the plan says `hr_self_rest`, but `_TOOL_REGISTRY["hr.apply_leave"]` is specced with CIBA scope `openid hr_self_rest` — confirm `hr_self_rest` actually grants *write* on the leave store, or this is a read scope being used for a write

**What:** `hr.apply_leave` is a *write* (it creates a leave request via `hr_service.apply_leave(...)` → `store.leave_requests[new_id] = {...}`). The plan (sprint-5.md §3 item 4, tech-arch §8, slice S5.1) gives it MCP scope `hr_self_rest` and CIBA scope `openid hr_self_rest` — the *same* scope `hr.read_balance` and `hr.read_history` use (both reads). Sprint 4 introduced the `*_write_rest` tier convention (`hr_assets_write_rest`, `it_assets_write_rest`) for exactly this — write-tier tools name a distinct scope so the CIBA token-C carries only what's needed and an audit reviewer can tell read from write at the scope level. `apply_leave` is "write your *own* leave request," which is arguably self-service rather than admin-write, so reusing `hr_self_rest` is *defensible* — but it should be a *documented decision* ("self-service writes stay on `hr_self_rest`; only cross-user/admin writes get a `*_write_rest` scope"), not a default. If `hr_self_rest` is currently a read-only scope on the WSO2 IS side, this won't even work — the MCP validator will pass it (it just checks scope membership) but the *intent* is muddied.

**Where:** `hr_server/mcp/tools.py` (new `apply_leave` handler, `required_scopes=frozenset({"hr_self_rest"})`); `hr_agent/ciba/orchestrator.py:_TOOL_REGISTRY["hr.apply_leave"]` (`"openid hr_self_rest"`); the new `hr.apply_leave` skill in `hr_agent_valid.json` (`"scope": "hr_self_rest"`); the IS scope catalogue (out of repo — confirm `hr_self_rest` is intended to cover self-service writes).

**Recommended fix:** Add one sentence to sprint-5.md §3 item 4 (or the api-design doc §2): "`hr.apply_leave` uses `hr_self_rest` because it is a self-service write (the user's own leave request); the `*_write_rest` tier is reserved for cross-user/admin mutations (`hr_assets_write_rest`, `it_assets_write_rest`)." If the council disagrees and wants `hr_self_write_rest`, that's a one-line `_TOOL_REGISTRY` + card + IS-catalogue change and S5.1 is the slice to do it in. Either way, write the decision down — otherwise Sprint 6's "persistent denylist" / audit work inherits an ambiguous scope.

### F-7 — LOW (non-blocking) — `GeminiLLMClient` builds *two* `ChatGoogleGenerativeAI` instances (router temp=0, composer temp=0.3) eagerly in `__init__` — fine, but the lazy-import boundary must be the *whole* `gemini.py` module, and `main.py`'s lazy `from orchestrator.llm.gemini import GeminiLLMClient` is the only import site — confirm nothing in `orchestrator/llm/__init__.py` pulls `gemini` transitively

**Recommended fix:** Add to the S5.2 slice checklist: "`orchestrator/llm/__init__.py` is empty (or re-exports only `client.py` symbols); `router.py`/`composer.py`/`chat/routes.py` import from `orchestrator.llm.client`, never `orchestrator.llm.gemini`; `gemini.py` is imported *only* by `main.py`, inside the `llm`-mode branch." Add a test: `import orchestrator.chat.routes` (and `orchestrator.main`'s import-time code, i.e. `create_app` without calling it) must succeed with `langchain_google_genai` *not* installed — this is the regression guard the code-reviewer leg will also want.

### F-8 — LOW (non-blocking) — the composer prompt instructs "use a bullet list only when listing 3 or more items; otherwise prose" — but several `_render_result` fallback fragments are *already* multi-line bullet lists (`hr.read_policy`, `hr.read_history`, `it.list_available_assets`, `it.get_my_assets`), so the LLM-mode reply and the fallback reply will look noticeably different for the same data; acceptable, but call it out so the manual-gate reviewer isn't surprised

In LLM mode the composer produces prose-or-3+-bullets per the §4c rules; in fallback mode the user gets the existing `_render_result` output, which for policy/history/asset-list tools is already a `"\n".join(["You can apply for these leave types:", "  • Annual Leave — …", …])` style block. Same data, two formats. That's *fine* for a demo (the §6 milestone-exit criterion is "SPA still works," not "byte-identical") — but the Stage-11 manual-gate runbook should note "the fallback reply formats lists differently from the LLM reply; both are correct" so the reviewer doesn't log it as a bug.

**Where:** `orchestrator/chat/routes.py:_render_result` (the existing multi-line bullet fragments); `orchestrator/llm/prompts.py` §4c (composer prompt list rule); `sprint-5-stage-11-manual-gate.md` (note to add).

### F-9 — LOW (non-blocking) — `render_outcomes` truncates `json.dumps(data)` at ~2 KB with `… (truncated)` — confirm no tool returns a `data` dict large enough to hit this in practice, and that truncation can't drop the *one field the composer needs* (e.g. `request_id` on `apply_leave`, `leave_id` on `approve_leave`)

**What:** Tech-arch §4d caps each outcome line at ~2 KB to keep the composer prompt bounded (R1) and avoid leaking an accidental blob. Sensible. But `json.dumps(data, default=str)` then truncated at a byte offset can land *mid-key* and drop trailing fields — and for the write tools the trailing fields are exactly the ones the composer prompt says "never invent" (`request_id`, `leave_id`, `cubicle_id`, `asset_id`). In practice none of the current tool results are anywhere near 2 KB (the biggest is `get_leave_history` with ≤10 entries × ~6 fields ≈ 1 KB worst case; `get_cubicle_summary` is 4 floors × 2 ints). So this won't fire today — but if it ever does, the composer is *required* to produce a reply that "states only facts present in the tool outputs" while the most load-bearing fact was truncated off.

**Where:** `orchestrator/llm/prompts.py:render_outcomes` (new, S5.2); the tool result shapes in `hr_server/mcp/tools.py` (`ApproveLeaveResult.request_id`, the new `ApplyLeaveResult.request_id`, `GetLeaveHistoryResult.entries`).

**Recommended fix:** (a) Raise the cap to ~4 KB (a `flash` prompt at 4 KB is still ~1 K tokens — negligible) *or* (b) better: instead of byte-truncating the JSON string, cap the *number of list elements* (`entries[:8]`, etc.) and serialise the trimmed dict — then it's always valid JSON and the scalar identity fields are never at risk. (b) is a few lines and strictly better; do it in S5.2 when `render_outcomes` is written. Either way, add a `test_render_outcomes_keeps_identity_fields_when_data_large` to the S5.2 list.

### F-10 — LOW (non-blocking) — `hr.lookup_employee` is in `_TOOL_REGISTRY` but is an *internal* step (the agent calls it to resolve a `sub` before `cubicle_assign`), not a chat tool the LLM should route to directly — the card expansion (§9) lists it as a skill, which means the router prompt will offer it to the LLM

**What:** §9 says "expand `hr_agent_valid.json` skills to cover every `_TOOL_REGISTRY` key: … `hr.lookup_employee` …" and the Stage-10 test enforces `card-skills == set(_TOOL_REGISTRY)`. But `hr.lookup_employee` returns `{found, username, email, sub}` — including the IS UUID — and the *only* legitimate caller is the HR agent itself, mid-`cubicle_assign`, to turn a username into a `login_hint`. There's no user message that should route to "look up an employee" as a standalone action; if the LLM picks it (because it's in the catalogue), the orchestrator runs a CIBA for `hr_read_rest`, the MCP tool returns the `sub` to the *agent*, the agent returns a `ResultPayload(data={found, username, email, sub})` to the *orchestrator*, and now `render_outcomes` puts `sub` in the *composer prompt* — a (small) widening of what reaches Gemini, contradicting §2.5 / §12 ("does not give the LLM … any `sub`"). The `LookupEmployeeResult` docstring already says "The SPA / chat surface NEVER sees this response — the agent transforms it." But the §9 card-expansion + §6 outcomes-collection would route it straight to the composer.

**Where:** `tests/fixtures/agent_cards/hr_agent_valid.json` (the new `hr.lookup_employee` skill per §9); `orchestrator/llm/router.py:_build_catalogue` (would include it); `orchestrator/chat/routes.py:_run_serial_fan_out` `outcomes` collection (would put its `data` — incl. `sub` — into a `ToolOutcome`); `hr_server/mcp/tools.py:LookupEmployeeResult` (returns `sub`).

**Recommended fix:** Two clean options, recommend the first: (a) **Don't put internal-only tools in the card.** Change the §9 contract from "`card-skills == set(_TOOL_REGISTRY)`" to "`card-skills == set(_TOOL_REGISTRY) - INTERNAL_TOOLS`" where `INTERNAL_TOOLS = {"hr.lookup_employee"}` is a small, explicit, commented set in the dispatcher (and the consistency test asserts that). This keeps `lookup_employee` reachable by the agent (it calls `mcp_client.lookup_employee` directly, not via the registry-dispatch path) — wait, check: the agent calls it via the *registry* (`getattr(deps.mcp_client, mcp_method)`)? No — `cubicle_assign`'s `kwargs_builder` takes `employee_username`/`employee_email` already; the agent does *not* currently call `lookup_employee` mid-assign in the shipped code (the `assign_cubicle` MCP handler takes username, not sub). So `hr.lookup_employee` is, today, a registry entry with *no caller at all* — which is its own minor smell. Either wire it (Sprint 5/6 if `cubicle_assign` should resolve+thread the sub) or drop it from `_TOOL_REGISTRY`. (b) If it stays in the catalogue: the §6 `ToolOutcome` mapping for `hr.lookup_employee` results must strip `sub` before building `ToolOutcome.data` — i.e., the orchestrator redacts it on the way to the composer, mirroring the SPA-side redaction the docstring already promises. (a) is cleaner; do that.
